# B-2.1 ‚Äî Federation user link + audit tenant scoping

Status: implemented (preview + Drizzle schema). Migration NOT yet run on real DB.

## Scope

Two additive changes that close the Federation Bridge ¬ß7‚Äì8 / ¬ß22‚Äì23 compatibility gap and prepare audit events for multi-tenant scoping (Constitution ¬ß7).

### 1. `federation_users` (new table)

Links a local `users.id` to one or more federated peers without changing the local identity row. Implements:

- `bbs_federation_user_id` ‚Äî cross-system identifier, unique per peer.
- `source_system_id`, `source_tenant_id`, `source_record_id` ‚Äî provenance (Fed ¬ß22).
- `data_origin` ‚Äî local | imported | synced | external.
- `sync_status` ‚Äî local | imported | synced | pending | conflict | archived_remote | disconnected (Fed ¬ß23).
- `sync_created_at` / `sync_updated_at` ‚Äî federation timestamps.
- `metadata` JSON ‚Äî peer-specific extras.

Constraints: `UNIQUE (peer_id, bbs_federation_user_id)`, indexes on `user_id` and `sync_status`.

### 2. `audit_events.tenant_key` (additive column)

`VARCHAR(64) NULL`, indexed. Allows tenant-scoped audit filtering without touching pre-existing event rows.

Audit hash-chain payload schema bumped from **v1** to **v2** (now includes `tenantKey`). The forthcoming `0003_audit_tenant_key` migration on real MySQL must insert an explicit `audit.chain.checkpoint` event (`from=v1, to=v2`) right after the ALTER. Preview mode emits this checkpoint as the first event so the verifier stays green on a fresh chain.

## UI changes

- `/registry/federation` ‚Äî peer list now selectable; selected peer shows a **Linked users** table (local user, federation UID, source system/tenant/record, origin, sync status). Two demo peers + two linked users are seeded so the screen is populated immediately.
- `/audit` ‚Äî new **Tenant** column and tenant filter dropdown above the table. Demo events are tagged with `tenant_key = "builder"`.

## Migration plan (NOT executed in B-2.1)

Future migrations to be generated when cPanel target is wired:

```sql
-- 0002_federation_users.sql
CREATE TABLE federation_users (...);
GRANT SELECT, INSERT, UPDATE, DELETE ON federation_users TO bbs_app;

-- 0003_audit_tenant_key.sql
ALTER TABLE audit_events ADD COLUMN tenant_key VARCHAR(64) NULL;
CREATE INDEX audit_events_tenant_key_idx ON audit_events(tenant_key);
INSERT INTO audit_events (...) VALUES (..., 'audit.chain.checkpoint', ..., '{"from":"v1","to":"v2","note":"prod-migration"}', ...);
```

Reversible: both via `DROP COLUMN` / `DROP TABLE`. Checkpoint row is retained.
