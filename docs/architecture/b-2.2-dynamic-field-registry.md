# B-2.2 — Dynamic Field Registry (DFR)

Status: implemented in preview (mock-store + UI). Real DB migration deferred per B-2 decisions.

## Tables (9)

Located under `src/db/schema/registry/fields/`:

| Table | Purpose |
|---|---|
| `field_definitions` | Field as object (head). `current_version_id`, `tenant_key`, lifecycle status. |
| `field_versions` | Per-edit version body (`schema` jsonb, validation, UI hints). |
| `field_translations` | Locale labels bound to a **version** (preserves old labels). |
| `field_taxonomy_bindings` | N-N field ↔ taxonomy node with kind `scope|filter|classifier`. |
| `field_permissions` | Per-field grants for role/group/user/federation_peer; `deny > allow`. |
| `field_entity_bindings` | Which Universal Profile Engine entities the field appears on. |
| `field_values` | EAV storage with typed columns + partial indexes + optional `position`. |
| `field_value_history` | Value-level audit + restore (1:1 to `audit_events`). |
| (enums) | `field_status`, `field_data_type`. |

## Lifecycle

`active → hidden → disabled → deprecated → archived`

- Reverse hops require admin + `change_reason`.
- Archived fields with any `field_values` row cannot be physically deleted.
- Lifecycle dashboard at `/registry/fields/lifecycle`.

## Permissions

Three layers (AND):

1. Module-level RBAC (`field.admin|editor|viewer`).
2. Per-field grants (`field_permissions`).
3. Entity-level permission from Universal Profile Engine.

`subject_kind=federation_peer` enables peer-scoped read-only grants via B-2.1 `federation_users`.

In-memory permission cache (`src/lib/field-registry.ts` → `resolveFieldPermissions`, `invalidatePermissionCache`). Invalidated on audit events tagged `field.permission.*` or status changes.

## B-2.2 open decisions — resolved

1. **Reference data_type** — IGEN. `schema.reference.entity_type` in `field_versions.schema`. Tenant-aware + permission-checked at read.
2. **Materialized projection** — PREPARATION ONLY. Interface + registry + hook (`registerProjection`, `listProjections`) live in `field-registry.ts`. Physical flat tables (`profile_vehicle_flat`, …) deferred to B-3.
3. **Multivalue position** — IGEN. `field_values.position INT NULL`, used for galleries, phone lists, social links, equipment lists, build timelines.
4. **Permission cache** — IGEN, simple in-memory Map. No Redis / distributed cache.

## Audit

All DFR mutations write to `audit_events` (B-2.1 v2 schema, hash-chained, `tenant_key`-scoped). Target types:

- `field_definition` (create / status change)
- `field_version` (new version)
- `field_translation`
- `field_permission`
- `field_value` (write)

Every `CREATE TABLE` in real migration MUST be followed by explicit `GRANT` for
`authenticated` and `service_role` (no `anon`), then `ALTER TABLE … ENABLE ROW
LEVEL SECURITY`, then `CREATE POLICY … USING (public.has_role(auth.uid(),
'admin'))` style. Per B-2 decisions, the migration runs in B-3, not now.
