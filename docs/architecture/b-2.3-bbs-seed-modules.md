# B-2.3 вАФ BBS Seed Module Catalogue

**Status:** implemented (preview-only, additive)
**Scope:** registry seed for all BBS modules. No business logic, no DB migration.

## Goal

Make every planned BBS module visible in the Project Modules registry from day
one, so the roadmap, dependency graph and lifecycle state are first-class data
вАФ not hidden in docs.

## Rules applied

- New modules ship with `state = 'planned'`.
- The B-1 foundation four (`identity`, `audit`, `project`, `rbac`) keep their
  current `in_dev` state вАФ this PR does not promote or demote anything.
- No business logic is added. Each module is only a registry row with:
  `key`, `category`, `state`, `visibility`, `roadmapPhase`, `dependsOn`.
- No real DB migration is executed. Drizzle schema for `project_modules`
  already exists from B-1 and is unchanged; the seed lives in the mock store.
- Every new user-visible string is rendered from i18n
  (`modules.catalog.<key>.{name,description}`,
  `modules.categories.<category>`, `modules.columns.*`).

The `dependsOn` array on each `MockModule` is the machine-readable form and
is rendered as badges in the Modules table.

## UI changes

- `/modules` table now shows: Key, Name (i18n), Category, Phase, State,
  Visibility, Depends-on.
- New category filter (`All | foundation | registry | profile | platform |
  portal | compliance`).
- Description below the name is i18n-driven; falls back to empty string if a
  locale is missing the key (no `[missing key]` text).
- Audit events for state transitions now correctly include `tenantKey:
  "builder"` (was implicitly null before).
